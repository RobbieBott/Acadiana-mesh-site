Not on Firmware 1.14+

Action Important

These repeaters were observed sending only 1-byte path hash adverts and were not observed forwarding any multi-byte packets. This likely means they are running firmware older than 1.14 and will silently drop 2-byte and 3-byte packets. Once the switch to 2-byte and 3-byte packets on Companion nodes is made, these repeaters will no longer forward traffic to help the mesh.

Operators should update to firmware 1.14 or later. Flash your device →

Note: Some repeaters in this list may already be on firmware 1.14+ with path.hash.mode left at the default of 0. They simply weren't observed forwarding a multi-byte packet during the data window. Updating and setting path.hash.mode to 1 or 2 will help get you off this list. We also recommend that users periodically set their companion nodes to use 2-byte or 3-byte and send a message in the #test channel. This will generate multibyte traffic and allow us to see which repeaters are forwarding the traffic, indicating that they're updated and multibyte capable.
Firmware Version Notes: There is a known bug in 1.14.0 where multibyte adverts are not sent automatically, even if you have run the set path.hash.mode command on your repeater. The multibyte adverts only go out when triggered manually. This means a repeater on 1.14.0 may still appear on this list even if it's set to send out multibyte adverts. Upgrading to 1.14.1+ is recommended. However, 1.14.1 introduces a new issue where rxgain is disabled by default, which can reduce receive sensitivity. Re-enable it after upgrading with: set radio.rxgain on

Clock Synchronization

Action Recommended

Repeater clocks that drift out of sync can cause packet deduplication failures, inaccurate network maps, and signature validation errors. We check clock health by comparing the timestamp embedded in each repeater's advert packet against the time the observer received it.
